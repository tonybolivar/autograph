# Autograph autofill coverage audit, 2026-05-13

Snapshot of per-ATS field coverage after a Playwright-driven harness pass.

Master profile now has **101 fields** across about, experience, education, employment, logistics, preferences, writing, and demographic sections.

## Global improvements landed in this pass

- Demographic radios fall back to "Prefer Not To Answer" / decline labels when the canonical profile value (race, gender, etc.) does not match any visible option.
- Radio fill now skips if a peer in the same name group is already checked, avoiding double-fill when decline labels and canonical values both match.
- AG_DECLINE_OPTION_LABELS expanded to include "Prefer not to disclose", "Prefer not to say", "Choose Not to Disclose", "I don't wish to disclose", and other common phrasings seen in Lever surveys and similar.
- Pronoun denormalizer added so `personal_pronouns` (e.g. "he/him") maps to the visible "He/him" / "He" / "Him" checkbox labels used by Lever and others.
- labelMatcher tokenization now treats parentheses and square brackets as word separators so labels like "Location (City)" match the city alias set.
- Greenhouse fillDropdown now uses a mousedown sequence and a keyboard fallback to open react-select menus that ignore plain click(), restoring the EEOC Gender/Race/Veteran/Disability dropdowns.

## Iteration 10 fixes (Recruitee Vue v-model retry)

- **Recruitee custom-question radios reverting after click**: Hispanic or Latino and the New York state-of-residence radio both had `data-ag-filled=true` but `el.checked=false` in the harness. The mousedown+mouseup+click on the label works for `candidate.gender` and `candidate.work_authorization` radios, but Recruitee's custom-question radios bound to `candidate.openQuestionAnswers` reactive state re-render after our click and reset `el.checked` back to false. fillCheckbox now dispatches the press sequence on the label, then on the input itself if still not checked. After the sync path it schedules two re-fires at 250ms and 800ms that detect Vue's revert and re-commit. CHECKED set in the harness went from {Male, Yes} to {Male, Hispanic or Latino, Yes, New York}. FILLED count holds at 13.

## Iteration 9 fixes (Date format reformat for masked inputs)

- **BambooHR Date Available filled wrong value**: the Fabric DatePickerInput is a masked text input with placeholder `mm/dd/yyyy`. The extension was writing the raw ISO profile value `2026-06-01`; only the first two chars got accepted, then coerced to `02`, leaving the field as `02/dd/yyyy`. Added `agReformatDateForInput` in `domUtils.js` which detects ISO `YYYY-MM-DD` profile values and rewrites them to the input's placeholder format. Supported formats: `mm/dd/yyyy`, `dd/mm/yyyy`, `yyyy-mm-dd`, `yyyy/mm/dd`, `mm-dd-yyyy`, `dd-mm-yyyy`, `mm.dd.yyyy`. HTML5 `type=date` inputs receive the ISO form natively. BambooHR Date Available now reads `06/01/2026`. The FILLED count is unchanged at 11 since the field was already counted with the wrong value, but the value is now actually valid for submission. This is a generic helper that will also apply to any future adapter whose date input uses a placeholder mask.

## Iteration 8 fixes (Jobvite resume detection)

- **Jobvite resume not detected**: Jobvite renders the attachment file picker as a body-level overlay div `.jv-add-attachment#attachmentDropdown` that is not an ancestor of the form's "Add Resume" h3. The previous `shouldFillResumeInput` walk from `file-input-0` up looked for an h3 as a direct child or grandchild of each ancestor and never found one, returning undefined. The `agIsResumeFileInput` fallback also failed since the file input has no name, id, or aria-label hinting at resume. The Jobvite prefillPass already clicks the Select button under Resume / CV h3 headers exclusively (it skips Cover Letter), so when the attachment overlay becomes visible (no `ng-hide` on `.jv-add-attachment`) and the page has at least one non-cover-letter Resume h3, the visible file input is by construction the resume slot. Return true in that case. Cover-letter file inputs stay inside an `ng-hide` wrapper until their section is opened, so they continue to skip correctly. Jobvite FILLED moved 7 to 8, resume.files.length 0 to 1.

## Closed-as-open: Rippling DIV[role=combobox] demographic dropdowns

The Rippling Lead Mechanical Engineer apply form has 5 demographic dropdowns rendered as `<div role="combobox" aria-labelledby="field-N-label">` for Gender (field-65), Hispanic/Latino (field-78), Veteran Status (field-84), Disability Status (field-90), plus an INPUT-based race combobox (field-71). The INPUT one fills via the existing isTextish branch. The DIV ones do not fill: even after gating on `isDropdown(el)` in content.js's profile-match condition, adapter.fillDropdown is never called for the DIVs. A trace `console.log` confirmed three calls for field-71 and zero calls for field-65/78/84/90. The manual mousedown + click sequence does open the menu and pick an option correctly when invoked from page.evaluate, so the mechanism works; something earlier in content.js's per-element loop skips these DIVs. Investigation reached three iterations without a successful fix, so the iteration was aborted per the loop discipline and content.js / rippling.js were restored.

Likely next-iteration angle: instrument content.js's loop iterator to log every element processed (or just the DIVs by tag filter) to confirm whether `els` includes the DIVs, whether `isExcluded` flags them, or whether `value !== undefined` short-circuits the path before fillDropdown.

## Iteration 6 fixes (Phenom UMI/USA, Rippling resume, Adobe Phenom tenant)

Three additional defects closed during the overnight push:

1. **Phenom country picks UMI over USA**: Adobe Phenom's country select offers `UMI` (United States Minor Outlying Islands) ahead of `USA` (United States of America) alphabetically. Profile country `United States` had no exact-text match against either, so `agFillSelect`'s startsWith fallback locked onto UMI (the first option whose text starts with "United States"). Added `country` and `phone_country` denormalizers that emit ordered candidates with `United States of America` first, then `United States`, then `USA`/`US`. The candidate loop now hits the longer specific text first and `agFillSelect` exact-matches USA in one pass. Country and Country Phone Code on Adobe now read `USA` / `USA_1`.
2. **Rippling resume not detected**: the resume file input sits inside a generic `<label>` whose text is `Drop or select (.doc / .docx / .pdf)`. The actual "Résumé*" heading lives in an Emotion-hashed div two ancestors up. Existing `agIsResumeFileInput` checked name / id / aria-label / direct label / class-regex; none of those hit on Rippling. Added an ancestor walk (up to 4 hops) that checks each ancestor's own text (excluding the subtree that already contains the input) for resume/résumé while bailing on cover letter. Regex now also accepts `r[ée]sum[ée]`. Rippling resume now fills (count 7 to 8).
3. **Adobe Phenom tenant unregistered**: `careers.adobe.com` is a classic Phenom site (`candidateSelfIdentifyAsPriorWorker`, `phoneWidget.countryPhoneCode`, `emailAgreement`, `smsOptIn` markers) but was not in `AG_KNOWN_CUSTOM_DOMAINS`. ## Iteration 3 fixes (user-driven follow-ups)

After round-2, the user flagged two real defects observed in production:

- TeamTailor was re-uploading the same resume on every mutation observer tick. Root cause: after the first upload, TeamTailor swaps the file input element for a fresh one; the new element is not in the WeakSet, so `fillResumeInputs` picks it up again. Fixed: `shouldFillResumeInput` now returns false when `candidate[resume_remote_url]` already holds an `https://` value, so we know the upload completed.
- Recruitee Hispanic-or-Latino radio (and similar Vue v-model radios) had `data-ag-filled=true` but `el.checked` reverted to false. Root cause: Vue listens for the bubbling click on the label wrapper and our `agSetCheckedSafe` only dispatched a plain `click()`. Fixed: Recruitee adapter now dispatches a `mousedown`+`mouseup`+`click` sequence on the radio's sibling/associated label. Confirmed: Hispanic or Latino, state of residence (New York), Male, sponsorship-yes all now check correctly.
- Eightfold privacy modal loop: user reported the Data Privacy modal re-appearing after every fill. Defaults investigation showed "Save my answers for future applications" is pre-checked by the page (defaultChecked: true, agFilled: false), so Autograph wasn't responsible for that. Mitigations added regardless: `waitForReady` delays first fill, modal/dialog containers are excluded from fill, consent/privacy/i-agree/acknowledge checkboxes are explicitly excluded, and `suppressRefillOnRerender: true` prevents re-fills during the page's React re-renders.
